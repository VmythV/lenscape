# 镜界 MVP 可行性评估

## 结论

当前 `docs/prd.md` 中定义的 MVP 可以实现。

第一版的关键不在复杂 AI 或实时视觉能力，而在于把拍摄指导内容做成稳定、可复用、可审核的模板系统。只要严格保持“拍摄前指导”的产品边界，不接入相机、不上传照片、不做拍后分析，MVP 的技术风险可控。

## 推荐实现方式

第一版建议采用以下方案：

1. 使用结构化配置维护拍摄风格、人物类型、场景和姿势模板。
2. 使用规则匹配完成风格、人物类型、场景到姿势模板的推荐。
3. 使用人工制作或 AI 生成后人工审核的静态示意图作为模板图。
4. 在模板图上叠加人物轮廓、站位框、构图线、机位箭头等视觉元素。
5. 指导文案以模板字段为兜底，运行时由 LLM 生成或润色，通过统一 provider 抽象接入。
6. 收藏与历史第一版以纯本地存储实现；分享、登录、后台和热更新作为后续能力。

## MVP 可实现范围

建议第一版控制在以下范围内：

1. 支持 6 到 8 个拍摄风格。
2. 支持男生、女生、不限性别、情侣、多人。
3. 支持 8 到 10 个常见场景。
4. 每个风格提供 3 到 5 个高质量姿势模板。
5. 每个模板包含示意图、人物轮廓图、文字步骤、机位说明和构图说明。
6. 用户完成“选择风格 -> 选择人物类型 -> 选择场景 -> 查看图文指导”的完整闭环。

## 不建议第一版实现

以下能力会显著提高成本和风险，不建议进入第一版：

1. APP 内拍照或录像。
2. 用户照片或视频上传。
3. 拍后评分和成片分析。
4. 实时人体骨架识别。
5. AR 站位线或实时导拍。
6. 自动修图、作品管理和社区。
7. 复杂会员体系和专业相机参数控制。

## 主要风险

1. 模板内容质量决定产品体验，需要人工审核和持续打磨。
2. 可视化模板资产制作量较大，需要控制第一版风格和模板数量。
3. 指导文案必须具体可执行，不能只输出泛泛建议。
4. 如果过早引入拍照、上传、评分或实时识别，产品边界会失控。

## 建议研发拆分

1. 数据层：风格、场景、人物类型、姿势模板配置。
2. 推荐层：根据风格、人物类型和场景筛选模板。
3. 展示层：首页、风格选择页、人物类型页、场景页、拍摄指导页。
4. 资产层：模板示意图、人物轮廓图、构图辅助图。
5. 可选能力：收藏、历史浏览、分享、后台维护。

## 待确认事项（已确认结论）

进入研发前的关键决策已确认：

1. 目标平台：**Flutter 跨平台**（iOS + Android 一套代码）。
2. 模板图：**AI 文生图 + 人工审核**。
3. 收藏与历史浏览：**第一版实现**，纯本地存储，无账号。
4. 登录账号：**第一版不做**。
5. 后台与热更新：**第一版不做**，模板随版本发布。

## LLM 与后端补充

第一版在静态模板地基之上接入 LLM，提供「风格推荐」与「指导文案生成」：

1. 增加一个**轻量后端代理**保管 API Key、做限流与缓存，客户端不直接持有 Key。
2. LLM 调用通过**统一 provider 抽象（工厂模式）**封装，支持 Anthropic 与 OpenAI 协议双实现，可按配置切换或降级。
3. LLM 不可用时，回退到本地静态模板文案，保证核心闭环始终可用。

详见 [docs/architecture.md](architecture.md) 与 [docs/llm-spec.md](llm-spec.md)。
